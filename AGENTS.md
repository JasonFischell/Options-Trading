# AGENTS.md

## Purpose

This repository is a Rust-first IBKR options automation project. Codex should optimize for small, well-verified increments that preserve paper-trading safety and minimize token usage across long-running development.

## Default Workflow

1. Inspect the current state before editing.
2. Prefer one narrow milestone per session.
3. Update only the subsystem needed for that milestone.
4. Run targeted verification before stopping.
5. Commit after every major change with an autogenerated commit message derived from the diff.

## Project Rules

- Prefer Rust for all production work.
- Treat `Python Code/` as a parity/reference harness only unless the user explicitly asks to change it.
- Avoid broad refactors unless the current phase requires them.
- Keep IBKR-specific code behind narrow adapter boundaries so scanner logic can be replay-tested without a broker connection.
- Use delayed, delayed-frozen, or frozen data for prototyping whenever possible.
- Favor snapshots and short-lived subscriptions over large streaming fan-out.
- Keep designs safely below IBKR market-data line limits.
- Default to dry-run or read-only behavior until the user explicitly asks to enable paper submission.
- Never enable live-money order flow by default.

## Session Efficiency

- Prefer vertical slices over large rewrites.
- Reuse existing modules and tests before introducing new abstractions.
- When adding configuration, include conservative defaults so future sessions do not need to re-plumb local setup.
- Summarize what changed, what was verified, and what the next best milestone is.
- Keep docs updates focused on architecture, workflow, or setup changes that materially affect future sessions.

## Windows Temp Directories

- Anticipate routine Windows file-lock issues for builds, tests, and other generated artifacts.
- Create per-run temp directories outside the repository under `%LOCALAPPDATA%\Temp`, using a project-specific prefix such as `options-trading-*` or `target-codex-*`, instead of placing temp outputs inside the worktree when lock contention is likely.
- Reuse those external temp directories for `CARGO_TARGET_DIR` and other scratch paths when that reduces lock conflicts.
- Remove temp directories when they are no longer needed, but never delete paths that were not created for the current task.
- Treat creation and cleanup of these temp directories as expected repository workflow, so Codex should do it proactively rather than pausing to ask the user for confirmation each time.

## Commit Guidance

- After each major milestone, stage the intended files and create a commit.
- Generate the commit message from the actual diff rather than using a placeholder.
- Keep commit scope tight enough that a later rollback or review is easy.

## When To Tell The User To Open A New Chat

Codex should explicitly recommend opening a new chat when:

- the current session has shifted to a clearly new milestone or subsystem
- the user is switching from planning/review to implementation, or from one major implementation area to another
- enough context has accumulated that continuing in the same chat is likely to waste tokens or reduce answer quality
- the next task is logically standalone, such as moving from scanner work to execution work, or from code changes to CI/debugging
- a previous branch of exploration is complete and the next step would benefit from a clean problem statement

When recommending a new chat, Codex should:

- say so directly and briefly
- suggest the exact next prompt the user can paste
- do this as a helpful efficiency recommendation, not as a blocker
