# Staged Implementation Plan With Logical Commits

## Summary

Implement this in bounded stages, each ending with targeted verification and a tight commit. Keep the system paper-only by default, snapshot-based, and Rust-first. Add a new configurable deployment budget with default `$10,000`, sourced from cash by default; `buying_power` will be available for analysis-only views, not routed paper sizing.

Recommended implementation order:
1. Wrapper/config foundation
2. Reporting/status improvements
3. Throughput optimizations
4. Pricing ladder and execution behavior
5. Cash allocator and multi-lot sizing
6. Safe cleanup and docs polish

## Stage 1: CLI + TOML wrapper + config model

Add a real CLI with:
- `scan --config <path>`
- `status --config <path>`

Add TOML config support with precedence:
1. CLI flags
2. TOML config
3. env vars
4. built-in defaults

Add/rename config inputs:
- ticker file or comma-separated tickers, with one required
- expiration dates in `YYYYMMDD`, at least one required
- min/max underlying price, default `$1` to `$20`
- min profit dollars per share, default `$0.05`
- min profit percent of initial investment, default `1%`
- min profit buffer percent, default `10%`
- deployment budget, default `$10,000`
- capital source selector: `available_funds` or `buying_power`
- enforce that `buying_power` is analysis-only
- concurrency and repricing settings
- per-symbol cash cap and reserve settings

Public/interface changes:
- `AppConfig` expands to support TOML and CLI-derived values
- add explicit config fields for `deployment_budget` and `capital_source`
- map current expiration-yield/downside-buffer internals to clearer user-facing names

Verification:
- config parse tests
- precedence tests
- required-input validation tests
- `cargo test -p ibkr-options-engine`

Commit scope:
- config loader, CLI entrypoints, config template, compatibility mapping

## Stage 2: Status command + at-a-glance reporting

Promote status into the main binary instead of a sidecar-only workflow.

Add `status` output that shows:
- account cash, buying power, net liquidation
- active deployment budget and selected capital source
- open orders
- open covered-call positions
- ledger state
- concise warnings when `buying_power` is selected for analysis only

Improve scan/report output to show:
- active thresholds and filters
- selected expirations
- deployment budget, reserve, allocatable capital, per-symbol cap
- top candidates
- concise allocation/proposal summary
- top rejection reasons
- normalized log artifact locations under top-level `logs/`

Verification:
- report rendering tests
- status output tests with replay data
- `cargo test -p ibkr-options-engine`

Commit scope:
- reporting model changes, `status` command, artifact normalization

## Stage 3: Bounded async throughput improvements

Refactor scan execution into bounded concurrency without long-lived streams.

Changes:
- switch market-data mode once per cycle, not per symbol
- add symbol-level concurrency with semaphore, default `4`
- add per-symbol option-quote concurrency, default `2`
- keep per-symbol request order intact
- preserve delayed fallback and invalid-contract skipping
- add cycle timing metrics and throughput counters

Keep safety constraints:
- no persistent fan-out
- no always-on subscriptions
- stay within conservative IBKR request behavior

Verification:
- replay-provider tests for deterministic results under concurrency
- timing/metrics population tests
- full test suite

Commit scope:
- scanner/market-data orchestration only

## Stage 4: Better combo debit pricing + bounded auto-reprice

Separate evaluation math from execution pricing.

Add pricing metadata to candidates/intents:
- stock bid/ask/reference
- option bid/ask/mid/model-fallback context
- natural combo debit
- midpoint combo debit
- profit-floor cap debit

Execution behavior:
- compute balanced start debit from midpoint, capped by profit floor
- compute ceiling debit from natural price, capped by profit floor
- if needed, auto-reprice through a small bounded ladder
- default to 3 attempts max with short waits
- stop on fill, rejection, or profit-floor breach

Keep routing rules:
- paper-only by default
- combo BAG only
- no live routing enablement

Verification:
- unit tests for ladder construction and stopping conditions
- executor tests for single-shot vs repriced flow
- full test suite

Commit scope:
- pricing model, order-intent pricing fields, executor repricing loop

## Stage 5: Cash allocator + deployment budget + multi-lot sizing

Replace the current affordability gate with cash-based allocation.

Allocation policy:
- use `available_funds` for routed paper sizing
- apply `deployment_budget`, default `$10,000`
- actual deployable amount is `min(available_funds, deployment_budget)` after reserve
- if `capital_source=buying_power`, use it for analysis-only allocation previews and logs, not routed orders
- collapse to best candidate per symbol before sizing
- allocate greedily by rank
- support one multi-lot combo intent per selected symbol
- enforce configurable max cash per symbol
- preserve existing open-symbol block safety

Type/model changes:
- `OrderIntent` gains lot quantity
- debit/credit/profit scale by lot count
- `CycleReport` gains allocation summary and capital-source details

Verification:
- allocator tests for budget cap
- reserve tests
- per-symbol cap tests
- multi-lot sizing tests
- analysis-only buying-power mode tests
- full test suite

Commit scope:
- allocator/state changes, intent quantity support, allocation reporting

## Stage 6: Safe cleanup + docs + lint clean

Remove only clearly unproductive legacy/dead surfaces while keeping the Python harness as reference.

Cleanup targets:
- remove unused `scoring` module if still unreferenced
- remove unused score-only model types if no longer needed
- remove stale `.env.example` knobs like `PROFIT_TAKE_PCT` and `MAX_LOSS_PCT`
- normalize generated log locations and `.gitignore`
- fix current Clippy findings
- update README to document the staged workflow, TOML config, `scan`/`status`, deployment budget, and capital-source policy

Verification:
- `cargo test -p ibkr-options-engine`
- `cargo clippy -p ibkr-options-engine --all-targets -- -D warnings`

Commit scope:
- dead code removal, doc updates, lint clean

## Assumptions and Defaults

- Default deployment budget: `$10,000`
- Default capital source: `available_funds`
- `buying_power` remains analysis-only, not routed paper sizing
- Python parity assets stay in the repo for reference
- GUI is out of scope for this milestone
- Each stage ends with its own logical commit derived from the actual diff
